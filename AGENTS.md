# AGENTS.md

本文件作用域覆盖：

- 当前仓库根目录
- 以及其所有子目录

## 子模块规则

处理某个模块、子项目或子目录中的文件前，必须先检查该范围内是否存在更具体的 `AGENTS.md`。

执行顺序要求：

1. 先读取本根目录 `AGENTS.md`
2. 再读取目标模块路径下最近的 `AGENTS.md`
3. 如果还有更深层的 `AGENTS.md`，继续按路径逐层收窄

规则优先级：

- 离目标文件越近的 `AGENTS.md`，优先级越高
- 子目录规则与父目录规则冲突时，以子目录规则为准
- 如果多个模块都会被修改，需要分别读取各自对应的 `AGENTS.md`
- 当前仓库应优先关注的分级目录包括 `apps/`、`packages/`、`docs/` 及其下的更深层子目录

## 根目录协作约定

- 文件放置应遵循“简洁、易懂、就近”的原则
- 新文件优先放到职责最匹配、最容易被后续维护者找到的目录
- 不要为了省事把脚本、数据、产物、文档直接堆在仓库根目录
- 如果某个文件明显属于某个模块，就放到该模块内，而不是放在公共区域
- 如果是跨模块复用的通用工具或说明，再考虑放在根目录公共位置
- 根目录下的通用脚本放在 `scripts/`
- 输入数据优先放在 `data/`
- 生成结果优先放在 `artifacts/`
- 临时文件统一放在 `tmp/`
- 通用说明优先放在根目录，模块专属说明优先放在对应模块目录
- 新增目录时，目录名应尽量一眼能看懂用途，避免含糊命名
- 不要跳过子模块自己的 `AGENTS.md` 直接修改其内部文件

## 文档阅读约定

- 处理某个领域的问题、设计或实现前，应先检查 `docs/` 下是否已有对应文档
- 如果 `docs/` 下存在与当前任务相关的文档，必须先阅读相关文档，再进行分析、设计、实现或修改
- 做技术栈、工程结构、架构边界相关工作时，优先阅读 `docs/tech-stack.md`
- 做产品 UI、设计系统、组件复用、页面模板、视觉统一性相关工作时，优先阅读 `docs/design-system/` 下对应文档
- 做模板初始化、项目改名、目录裁剪、启动约定相关工作时，优先阅读 `docs/template/` 下对应文档
- 如果代码现状与文档约定冲突，不能直接忽略文档；应先明确是按文档修正实现，还是先更新文档再修改代码
- 新增某个领域的重要约定后，应优先沉淀到 `docs/`，避免规则只存在于临时对话中

## 项目定位

- 整体协作原则保持“重后端、轻前端”
- 当前阶段优先做 `TypeScript` 全栈 MVP，产品层先保持最小接入面，后续再展开 `Web MVP`、`iOS`、小程序或桌面端
- `TypeScript` 侧的包管理、运行时与 workspace 工具统一以 `Bun` 为准，后续命令示例优先使用 `bun`
- 如果任务涉及数据清洗、离线数据处理、批量数据转换等场景，才考虑引入 `Python`
- 使用 `Python` 时，包管理工具和项目管理工具统一使用 `uv`，不使用 `poetry` 和 `pip`
- `PostgreSQL` 是固定数据库，建模与接口设计应优先围绕可复用的共享能力展开
- 不预设团队协作流程，不主动引入面向多人团队的重型规范、审批链路或仪式

## 架构协作约定

- 任何新增能力，优先先想清楚 `agent runtime` 的领域模型、状态流转、接口契约和可观测性，再考虑产品层交互形态
- 相关实现应与通用业务服务、基础设施访问、`TS` UI 展示逻辑分层放置，避免耦合
- 不要把 prompt、graph 编排、领域规则、数据访问逻辑混写在同一个大文件中
- `agent runtime` 的节点职责应保持单一：单个节点尽量只做一步明确决策、转换或调用
- `agent runtime` 的 `state schema`、节点输入输出、边的流转条件应保持可读、可追踪、可测试
- 工具调用、模型调用、记忆读写、外部系统访问应尽量通过明确边界封装，避免在业务流程里散落实现细节
- 若新增 agent 能力，优先复用已有状态定义、工具封装与领域服务，避免重复造轮子
- 重要流程应优先支持日志、追踪、调试信息或可观测性，便于排查 agent 行为
- `Python` 若被引入，应限于数据清洗、数据处理或离线脚本，不承载默认的线上 `agent runtime`、API 主链路或核心产品编排

## 目录与模块建议

- 如果项目开始进入多端协作，优先采用“端”和“共享包”分离的结构，而不是把所有代码混放
- `TypeScript` 产品应用应放在清晰的应用目录中，例如 `apps/` 下按端拆分
- 跨端复用的 schema、类型、领域模型、`TS` SDK、共享 UI 规范、tokens 应优先放在 `packages/` 或等价共享目录
- 数据库访问层、外部 API 适配层、模型供应商适配层应与业务编排分开
- `TS` 产品页面、组件、样式资源不要反向承载核心业务判断；业务规则应尽量收敛到共享层或后端
- `apps/` 下优先放具体应用与部署单元
- `packages/` 下优先放跨应用复用的共享能力
- `docs/` 下优先放架构、设计系统、流程规范与决策记录
- `Python` 辅助脚本如确有必要，应放在语义清晰的独立目录中，并与主 `TypeScript` 运行时边界明确分离

## 前后端权重约定

- 涉及取舍时，优先保证后端结构正确、领域边界清楚、接口稳定
- `TS` 产品实现以最小够用为优先，避免过早投入大量时间在复杂视觉效果、动画或重型状态管理上
- 若一个需求可以通过后端或 `agent runtime` 补充结构化数据来简化产品层，就优先调整后端，而不是在前端堆复杂推导逻辑
- 多端场景下，展示层允许各端适配，但接口语义、领域规则和核心行为应尽量一致

## 业务建模约定

- 领域命名应优先贴近当前产品的核心概念，而不是沿用模板默认名词
- 新增模型或表结构时，应先判断其属于用户长期资料、流程状态、执行记录、分析产物还是系统配置，避免概念混淆
- AI 产品中应尽量区分“用户确认数据”“计划/配置”“执行记录”“自动推断/建议”四类信息，避免混在同一字段或对象中
- 与任务流、流程流或状态机相关的状态流转应明确，例如待开始、进行中、已完成、已取消、已归档等，避免状态语义含糊
- 涉及用户偏好、记忆或模型推断时，应优先考虑可解释性和后续可维护性，不要只依赖隐式 prompt 行为

## 单开发者协作约定

- 默认协作形态是一个开发者配合 GPT-5.4 在 Codex 内持续开发
- 规则设计应优先服务“可发现、可复用、可让 AI 快速读懂”，而不是服务多人交接流程
- 没有明确收益时，不要主动增加重型流程、冗长模板或形式化文档
- 代码、文档与目录命名都应尽量让后续 AI 在最少上下文下也能快速定位

## 实现与测试约定

- 新增后端能力时，优先补领域层、graph 层或接口层测试，不要只做手工验证
- 修改 graph 编排、工具选择或状态字段时，应同步检查是否影响已有节点、边条件与持久化结构
- 若实现依赖模型输出，尽量通过结构化输出、schema 校验或防御式解析降低脆弱性
- Prompt 调整如果会影响行为边界，应在对应模块旁保留简短说明，便于后续维护者理解
- 前端改动如只是配合后端能力，应尽量小步提交，避免在同一次任务中引入无关 UI 重构

## 删除操作约定

- 仅当删除操作是用户明确提出的要求时，才适用本节规则
- 对于用户明确要求的删除操作，执行前必须向用户再次确认，不得跳过
- 确认时必须重新核对删除目标、影响范围和执行必要性
- 涉及文件删除时，必须先整理并再次核对待删除文件 list，得到用户确认后才可执行删除
- 本节不适用于执行任务过程中由代理自行创建并自行清理的项目文件、临时文件或其他辅助文件
