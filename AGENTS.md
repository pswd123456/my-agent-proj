# AGENTS.md

作用域：仓库根目录及全部子目录。

## 使用方式

- 修改任何文件前，先读根目录 `AGENTS.md`，再沿目标路径逐层查找更近的 `AGENTS.md`
- 越靠近目标文件的 `AGENTS.md` 优先级越高；若多个模块都会修改，分别检查各自路径
- 重点检查 `apps/`、`packages/`、`docs/` 及其更深层目录中的 `AGENTS.md`

## 通用协作规则

- 前端文案不要暴露实现意图、设计取舍或本次修改背景；只有当说明对用户真实有帮助时才添加
- 文件与目录命名保持简洁、贴职责、便于后续定位；不要直接照搬需求标题或文档标题
- 新文件遵循“就近放置、按职责归档”：脚本放 `scripts/`、输入数据放 `data/`、产物放 `artifacts/`、临时文件放 `tmp/`
- 不要把明显属于某个模块的文件堆到仓库根目录；跨模块复用内容再放公共位置
- 重要长期规则优先沉淀到 `docs/`，不要只留在临时对话中

## 开工前先看文档

- 处理某个领域前，先检查 `docs/` 是否已有对应文档
- 做架构、工程结构、技术栈相关工作时，优先看 `docs/tech-stack.md` 与相关架构文档
- 做 UI、设计系统、页面模板相关工作时，先看根目录 `DESIGN.md`，再看 `docs/design-system/`
- 做模板初始化、项目改名、目录裁剪、启动约定相关工作时，先看 `docs/template/`
- 若代码现状与文档冲突，先明确是修实现还是修文档，不能直接忽略文档

## 项目与技术取向

- 当前以“重后端、轻前端”的 `TypeScript` 全栈 MVP 为主，产品层保持最小接入面
- `TypeScript` 侧统一使用 `Bun`
- 仅在数据清洗、离线处理、批量转换等场景引入 `Python`；使用时统一采用 `uv`
- 默认数据库为 `PostgreSQL`
- 不主动引入面向多人团队的重型流程、审批链路或仪式

## 架构约定

- 新增能力时，先明确 `agent runtime` 的领域模型、状态流转、接口契约与可观测性，再设计产品交互
- 保持分层清晰，不要把 prompt、graph 编排、领域规则、数据访问混在同一大文件中
- 构建 prompt 时要兼顾整体 cache 稳定性：稳定且高复用的内容尽量前置，越容易变动的内容越往后放；按“前 n 个字符完美匹配”理解 cache 前缀命中，避免前部易变内容破坏后续稳定内容的 cache
- 优化工具描述、工具契约或模型引导时，要从产品级行为抽象问题：例如“局部文案编辑应只改目标内容并保留周边结构/控制流”。不要把单个 trace 暴露出的框架、语言或语法细节写成通用规则；具体语法只能作为测试 fixture、解析器能力或错误案例证据，不应成为工具描述的主要边界
- `agent runtime` 节点应单一职责，状态、输入输出、边条件应可读、可追踪、可测试
- 工具调用、模型调用、记忆读写、外部系统访问应通过明确边界封装
- 优先复用已有状态定义、工具封装与领域服务，避免重复造轮子
- 收紧旧有实现时，优先把真实调用面收敛到单一实现，并删除或改直旧路径；不要通过往上搭兼容层来保留旧代码，增加后续维护和审计成本。确需短期兼容时，必须写明迁移窗口、退出条件与清理计划
- `Python` 不承载默认线上 runtime、API 主链路或核心产品编排

## 目录分层原则

- `apps/` 放具体应用与部署单元
- `packages/` 放跨应用复用的共享能力
- `docs/` 放架构、设计系统、流程规范与决策记录
- 展示层不要反向承载核心业务判断；业务规则优先收敛到共享层或后端

## 实现与测试

- 新增后端能力时，优先补领域层、graph 层或接口层测试，不只做手工验证
- 修改 graph 编排、工具选择或状态字段时，同步检查节点、边条件与持久化结构
- 依赖模型输出时，优先使用结构化输出、schema 校验或防御式解析
- 前端若只是配合后端能力，尽量小步修改，避免顺手做无关 UI 重构

## 数据库迁移工作流

- 只要新增、修改或生成了迁移文件，必须在同一轮工作里完成“生成/检查 -> 执行 -> 验证 live DB -> 测试”的闭环；不能只写迁移，也不能只相信 CLI 成功提示
- schema 变更优先通过 `packages/db/src/schema.ts` 和仓库标准命令生成迁移；手写迁移时必须说明原因，并同时检查 SQL、`packages/db/migrations/meta/_journal.json`、快照文件是否一致
- 执行迁移前，先检查新迁移 SQL 是否符合预期：列的 `NULL/NOT NULL`、默认值、数据回填、索引/约束、可重复执行风险，以及是否会破坏现有数据
- 执行迁移前，必须确认 `packages/db/migrations/meta/_journal.json` 中新增条目的 `idx`、`tag` 和 `when` 与迁移文件顺序一致；`when` 不能早于已经存在的后续迁移，避免 Drizzle 漏跑新迁移
- 当场从仓库根目录运行标准命令：`bun run db:migrate`
- 迁移命令完成后，必须用真实数据库验证效果，不要只看 `migrations applied successfully`：检查 `drizzle.__drizzle_migrations` 账本，并用 `information_schema` 或 `pg_catalog` 验证新增/修改的表、列、索引、约束、nullable/default 等真实状态
- 迁移后运行 `bun run db:check`，并补跑与变更相关的最小测试集合；涉及 repository、API、worker 或 agent runtime 的 schema 变更，要跑对应包的定向测试
- 如果 `bun run db:migrate` 显示成功但 live schema、迁移账本或 `_journal.json` 不一致，必须立即停下来修复迁移链路；不能继续把问题归因到业务代码、模型或前端
- 最终回复里要说明：改了哪些迁移文件、实际执行了哪个数据库命令、live DB 验证了哪些字段/约束、跑了哪些测试；如有未验证项必须明确写出

## 删除操作

- 只有在用户明确要求删除时，才进入删除流程
- 执行删除前，必须再次向用户确认删除目标、影响范围与必要性
- 删除文件前，先整理待删除文件清单并获得确认
- 代理自行创建的临时文件可自行清理，不受本节限制
